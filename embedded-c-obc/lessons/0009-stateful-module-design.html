Lesson 0009: Stateful Module Design

A stateful embedded module remembers facts between calls without hiding ownership. This is the bridge between a pure decision helper and a flight state machine.

The One Thing

When a module needs memory between calls, make that memory explicit with a caller-owned context struct and a small init/update API.

OBC habit: avoid hidden global state by default. The caller provides storage, the module initializes it, and only the module mutates it through its API.

API Shape

typedef struct {
    FlightPhase phase;
    uint32_t samples_in_phase;
    bool fault_latched;
} FlightFsm;

void flight_fsm_init(FlightFsm *fsm);
FlightFsmDecision flight_fsm_update(FlightFsm *fsm,
                                    const FlightFsmInput *input);
PieceMeaning
FlightFsm *fsmCaller-owned module state. The module may modify it.
const FlightFsmInput *inputRead-only snapshot for this update.
flight_fsm_initPlaces the context into a known starting state.
flight_fsm_updateConsumes one input sample and advances module state if conditions are met.

Why This Matters

Flight code is reviewed under pressure. A context struct makes ownership inspectable: tests can create two independent instances, reset one without touching the other, and replay sample sequences deterministically.

Review smell: if update behavior depends on a file-scope variable that tests cannot reset, the module has hidden state.

Where The Loop Lives

The module does not loop by itself. The caller owns the loop: a test, scheduler, or later a FreeRTOS task calls flight_phase_update(...) once for each new sensor sample. The same FlightPhaseTracker is passed every time, so counters inside the context remember what happened on earlier calls. 

FlightPhaseTracker tracker;
flight_phase_init(&tracker);

for each new sensor sample:
{
    FlightPhaseInput input = build_input_from_sample(sample);
    FlightPhase phase = flight_phase_update(&tracker, &input);
    publish_or_log_phase(phase);
}

In the exercise tests, the loop is written out as repeated calls so you can see each transition:

CallInput ideaExpected phaseWhy state matters
1motor burning, climbingBOOSTThe tracker leaves idle.
2motor off, still climbingCOASTThe tracker remembers it was in boost.
3descending sampleCOASTOne descent-like sample is only a candidate.
4descending sample againDESCENTThe candidate counter persisted across calls.

Exercise Transition Contract

For this exercise, use this deliberately small set of phase rules. These are not a complete rocket flight detector; they are a teaching contract for practicing stateful module design.

Current phaseInput conditionNext phaseCounter behavior
IDLE motor_burning == true BOOST Reset samples_in_phase to 0 when the phase changes.
BOOST motor_burning == false COAST Reset samples_in_phase to 0 when the phase changes.
COAST vertical_velocity_mps < -10 for two consecutive samples DESCENT Increment descent_candidate_samples; transition when it reaches 2.
DESCENT altitude_m <= 5 and vertical_velocity_mps >= -1 LANDED Reset phase counters on the transition.
Any phase sensors_valid == false or null pointer input FAULT Fault is immediate.
Why two descent samples? one negative velocity sample might be noise. Requiring two consecutive samples demonstrates why state must persist between calls.

Stateful But Not Yet Event-Driven

Lesson 0009 still uses samples directly. For example, "two descending samples in a row" can confirm descent. That teaches state ownership before adding explicit events in lesson 0010.

if (input->vertical_velocity_mps < DESCENT_VELOCITY_MPS)
{
    fsm->descent_samples += 1u;
}
else
{
    fsm->descent_samples = 0u;
}

Practice

Open exercises/0009-stateful-module-design/. Implement a small flight-phase tracker with a caller-owned context struct.

  1. Keep public types and prototypes in flight_phase.h.
  2. Initialize every context field in flight_phase_init.
  3. Call flight_phase_update once per sample; do not write a loop inside the module.
  4. Update state only through flight_phase_update.
  5. Run rtk make test.
