Lesson 0001: volatile Is For Hardware Visibility

In OBC firmware, volatile is a precise tool: it tells the compiler that a load or store matters because something outside normal C code may observe or change the object. Today you will learn the one correct mental model and apply it to a tiny register driver.

The One Thing

A non-volatile object is controlled by the C abstract machine. The optimizer may keep it in a register, remove repeated reads, or remove stores whose results are never used.

A volatile-qualified access is an observable side effect. For memory-mapped I/O, this is what prevents the compiler from silently turning repeated register polls into stale cached values.

Mission rule: use volatile for hardware registers. Use synchronization primitives for communication between tasks and ISRs.

OBC Example

Imagine a UART status register where bit 0 becomes one when a received byte is ready. The register can change because the peripheral changes it, not because your C code assigns to it. 

#include <stdint.h>

typedef struct {
    volatile uint32_t STATUS;
    volatile uint32_t DATA;
} UartRegs;

enum {
    UART_STATUS_RX_READY = 1u << 0
};

uint8_t uart_read_byte(UartRegs *uart)
{
    while ((uart->STATUS & UART_STATUS_RX_READY) == 0u) {
        /* Real flight code should use a timeout, not an infinite wait. */
    }

    return (uint8_t)(uart->DATA & 0xffu);
}

The volatile members mean every read of STATUS and DATA must be emitted as an actual access. The pointer itself does not need to be volatile here; the pointed-to registers do.

Quick Check

1. A sensor driver's status register changes without a C assignment. What should be volatile?

2. A FreeRTOS ISR sets a flag and a task polls it. Is volatile bool flag a complete design?
