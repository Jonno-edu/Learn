The one thing

In embedded C, a declaration is not just a type. It is a contract between the program, the compiler, the CPU, the linker, the hardware, and sometimes an RTOS.

Core habit: every declaration should answer these questions: What is the value’s role? What range is valid? Who owns the storage? How long does it live? Can hardware change it? Can another execution context touch it? Is its binary representation visible outside the program?

Beginner layer: declaration roles

Start here. Before worrying about ABI, padding, strict aliasing, or FPU context save, learn to classify declarations by job.

Data byte uint8_t packet byte, checksum byte, raw register byte a size or loop counter
Size / index / capacity size_t frame_len, dst_cap, i a protocol byte field
Signed physical value int16_t, int32_t raw accelerometer count, temperature in milli-degrees unsigned register flags
Fractional physical value float or fixed-point integer calibrated voltage, normalized vector, filter coefficient blind use of double
Address T * const uint8_t *frame actual storage
Output storage T *out ObcPacket *out returning a pointer to a local variable

Simple rule: address + length is the default shape for buffer APIs: const uint8_t *buf, size_t len. Address alone is not a buffer.

Full C type map for firmware

This table is intentionally dense. It is the reference section you come back to when a declaration feels wrong.

Tool Use in firmware Limit / trap Habit
char Text and C strings. Plain char signedness is implementation-defined. Do not use it for binary bytes. Use char for text, uint8_t or unsigned char for bytes.
signed char, unsigned char Small integer values; object representation access through character types. Not the same semantic role as text. Be explicit when signedness matters.
short, int, long, long long General integer arithmetic where exact external width is not required. Widths vary by implementation and ABI. int is often natural for CPU arithmetic but not a wire type. Use for local calculations only when range is obvious and representation is not externally visible.
uint8_t, uint16_t, uint32_t, uint64_t Exact-width unsigned wire fields, registers, packed values, CRCs, masks. Exact-width typedefs are optional; they exist only if the implementation has a matching type. uint8_t may behave like a character type in formatted output. Use when bit width is part of the contract.
int8_t, int16_t, int32_t, int64_t Exact-width signed protocol fields and sensor raw values. Signed overflow is not a safe wraparound mechanism. int8_t may print as a character unless promoted/cast. Use signed types only where negative values are valid.
uint_leastN_t, int_leastN_t At least N bits, smallest available type. Useful for portability, but the actual width may exceed N. Use when you need a minimum range, not exact storage.
uint_fastN_t, int_fastN_t At least N bits, chosen for speed on the implementation. Can be much wider than expected, which can bloat arrays/structs. Bad for packet layout. Use for local arithmetic, not storage format.
size_t Object sizes, buffer lengths, capacities, indexes. Unsigned. Backward loops and signed comparisons can break. Use for memory extents; check before adding sizes.
ptrdiff_t Difference between two pointers into the same array object. Pointer subtraction outside the same array is not a portable measurement. Use for cursor differences; keep the original buffer bounds.
intptr_t, uintptr_t Integer type capable of holding a data pointer when provided by the implementation. Optional. Integer-to-pointer conversion is still platform-sensitive. Does not make arbitrary addresses valid objects. Use uintptr_t for register base constants and address arithmetic when necessary.
bool True/false state and predicates. Not an error-reporting system. Size is implementation-defined. Use bool only when two outcomes are enough.
enum Modes, states, command IDs, error codes. Underlying representation is implementation-defined enough that you should not expose raw enum storage on the wire. Encode/decode enum values explicitly at boundaries.
float Single-precision fractional math, filters, calibration, control algorithms, sensor conversion. Cost depends on FPU, ABI, libraries, and interrupt context. Decimal values such as 0.1 are not exact in binary. Use deliberately; suffix constants with f: 3.3f.
double More precision/range where required by the algorithm or host-side model. Often more expensive than float; sometimes 64-bit, sometimes target/ABI-dependent. Unsuffixed floating constants are double. Do not use by habit on small MCUs.
long double Rare in MCU firmware; high precision on some platforms. Representation and cost vary widely. Usually not appropriate for small embedded targets. Avoid unless your platform and requirement justify it.
Fixed-point integers Scaled units: mV, mA, m°C, Q15/Q31 DSP values. Scaling and overflow become your responsibility. Name the unit in the variable: battery_mv, temp_mdeg_c.
Pointers Addresses of objects, buffers, registers, callbacks, and opaque handles. A pointer does not know length or ownership. Invalid alignment can fault or misbehave. Pair buffer pointers with length/capacity and ownership rules.
Arrays Actual contiguous storage. Function parameters declared as arrays adjust to pointers. sizeof inside the callee does not recover the caller’s array length. Use sizeof array / sizeof array[0] only where the array object is still in scope.
struct Internal grouping of related fields. Padding, alignment, and endian make it unsafe as a raw wire/register layout unless carefully controlled. Good for internal objects; decode external bytes into it.
union Variant storage, register overlays in vendor headers, carefully controlled type punning patterns. Active member and aliasing rules are easy to misuse. Use sparingly; prefer explicit encode/decode for packets.
Bit-fields Readable local flags in internal structs. Allocation order, straddling, and signedness are implementation-defined; poor fit for wire formats and many registers. Use masks/shifts for hardware and protocols.
Function pointers Callbacks, driver interfaces, state-machine actions. Harder to trace statically; ISR safety and lifetime of context pointers matter. Pass a void *context with callbacks when state is needed.
void * Opaque generic data pointer. Type information is gone; alignment and lifetime still matter. Use only at abstraction boundaries, then cast once near the boundary.

Integer model and conversion traps

C integer behavior is not “small integers do small math.” Integer promotions and usual arithmetic conversions happen before many operations. This is where many embedded bugs hide.

Promotion before arithmetic

uint8_t a = 250u;
uint8_t b = 10u;

/* a and b are promoted before addition. */
uint8_t c = (uint8_t)(a + b);

The cast may be intentional modulo behavior, but it should be obvious and documented.

Check size addition

if (payload_len > SIZE_MAX - HEADER_LEN) {
    return ERR_OVERFLOW;
}
size_t total = HEADER_LEN + payload_len;

Unsigned wrap is defined, but usually still a bug in length/capacity calculations.

Shift with width discipline

if (bit >= 32u) {
    return ERR_BAD_BIT;
}
uint32_t mask = UINT32_C(1) << bit;

Shift counts must be valid, and signed shifts are a common source of undefined or surprising behavior.

Use <inttypes.h> for printing

#include <inttypes.h>

uint32_t ticks = 1234u;
printf("ticks=%" PRIu32 "\n", ticks);

Fixed-width integer typedefs are not guaranteed to match a particular printf format like %lu.

Integer checklist

  • Use unsigned types for masks and modulo counters where wrap is intentional.
  • Use signed types for physical values that can be negative.
  • Do not mix signed values and size_t without thinking through the comparison.
  • Do not cast away warnings; prove range first.
  • Do not use uint8_t as a loop counter just because the buffer is small.
  • Use named limits: UINT8_MAX, INT16_MIN, SIZE_MAX.

Floating point and fixed-point

Floating point is not forbidden in embedded C. It is a design choice. On an MCU with a correctly enabled single-precision FPU, float can be completely reasonable. On an MCU without an FPU, the same expression may become a software library call. With an RTOS, FPU use can affect context switching. With a strict ABI, object files compiled with incompatible floating-point options may not link safely.

Representation Good for Limit
float Calibration, filters, control math, normalized sensor values, readable formulas. Rounding, non-exact decimal fractions, possible FPU/library/context cost.
double Algorithms that truly need more precision/range. Unsuffixed constants and many math functions default toward double. Can be expensive.
Scaled integer Physical values with fixed units: mV, mA, m°C, ppm, ticks. You manage scale, rounding, saturation, and overflow.
Q-format DSP/control math with fixed binary scale, e.g. Q15 or Q31. Requires discipline; easy to multiply into overflow.

Decision rule

Use float when the equation is clearer and the target can afford it. Use fixed-point when determinism, storage, timing, wire format, or lack of FPU matters more. 

/* Fixed-point physical units. */
uint16_t battery_mv = 3700u;       /* 3.700 V */
int32_t temp_mdeg_c = 25375;       /* 25.375 °C */

/* Float physical value. */
float battery_v = 3.700f;

Floating-point traps in firmware

Unsuffixed constants

float v = adc * 3.3 / 4095.0;   /* double constants */
float w = adc * 3.3f / 4095.0f; /* float constants */

Use f suffixes when you want single-precision arithmetic.

Float loop counters

/* Bad: may miss, repeat, or not terminate as expected. */
for (float x = 0.0f; x != 1.0f; x += 0.1f) { }

/* Better. */
for (size_t i = 0; i <= 10u; ++i) {
    float x = (float)i / 10.0f;
}

Printf cost

Many embedded C libraries omit floating-point printf by default or pull in a large formatting path when enabled. 

/* Often cheaper for logs: */
printf("%" PRIu16 " mV\n", battery_mv);

FPU and ABI

On Arm/GCC, soft-float and hard-float ABI choices are not link-compatible. Treat floating-point compiler flags as a project-wide interface, not a file-local preference.

Fixed-point patterns

/* Convert ADC counts to millivolts without float.
   Example: 12-bit ADC, 3300 mV reference. */
static uint16_t adc_to_mv(uint16_t adc_count)
{
    uint32_t mv = ((uint32_t)adc_count * 3300u + 2047u) / 4095u;
    if (mv > UINT16_MAX) {
        mv = UINT16_MAX;
    }
    return (uint16_t)mv;
}

/* Q15 multiply with rounding: result is also Q15. */
static int16_t q15_mul(int16_t a, int16_t b)
{
    int32_t prod = (int32_t)a * (int32_t)b;     /* Q30 */
    prod += (int32_t)1 << 14;                  /* round */
    prod >>= 15;                               /* back to Q15 */

    if (prod > INT16_MAX) { return INT16_MAX; }
    if (prod < INT16_MIN) { return INT16_MIN; }
    return (int16_t)prod;
}

Memory layout, alignment, padding, and aliasing

C objects have representation and alignment. Firmware often violates those rules accidentally because packets, registers, DMA buffers, and flash images look like “just bytes.” They are not all the same thing.

Struct padding is normal

typedef struct {
    uint8_t  id;
    uint32_t timestamp;
    uint16_t value;
} Sample;

/* This is probably larger than 7 bytes because of padding. */
Do not serialize with memcpy(&wire, &sample, sizeof sample) unless the binary layout is explicitly controlled and tested for your compiler, ABI, and target.
Do not cast unaligned bytes to wider pointers. Some MCUs fault on unaligned access; others allow it slowly; others silently produce multiple bus cycles.

Packed structs are not magic

Compiler-specific packing attributes can remove padding, but they do not solve endian order, atomicity, alignment cost, strict aliasing, or future portability. Packed structs are sometimes appropriate for carefully reviewed hardware or wire definitions, but explicit encode/decode helpers are usually safer for protocol code.

Strict aliasing and byte access

Accessing object representation through character types is a special case in C. Reinterpreting arbitrary object pointers as unrelated struct or integer pointer types is not a general-purpose decoding strategy. 

/* Good: decode bytes explicitly. */
static uint32_t read_be32(const uint8_t b[4])
{
    return ((uint32_t)b[0] << 24) |
           ((uint32_t)b[1] << 16) |
           ((uint32_t)b[2] <<  8) |
           ((uint32_t)b[3]);
}

/* Risky: alignment, aliasing, and endian assumptions. */
uint32_t x = *(const uint32_t *)&frame[0];

Bit-fields

Bit-fields look tempting for registers and packets, but their allocation order, straddling behavior, and some signedness details are implementation-defined. Use masks and shifts for hardware and wire formats unless your vendor header or coding standard gives a specific, reviewed reason.

Wire formats, endian, and serialization

External binary representation is a contract. Internal C object layout is an implementation detail. Keep them separate.

Boundary Preferred representation Reason
UART/SPI/I2C/CAN packet uint8_t byte stream + explicit decode Controls endian, length, checksum, and field validation.
Flash settings image Versioned byte layout + CRC + defaults Survives firmware updates and struct changes.
Internal application object struct Readable, type-safe grouping after validation.
Network-style integer Explicit big-endian/little-endian helpers Avoids host-endian assumptions. 
typedef struct {
    uint8_t msg_id;
    uint16_t sensor_raw;
    uint16_t battery_mv;
} Telemetry;

static bool telemetry_decode(const uint8_t *buf,
                             size_t len,
                             Telemetry *out)
{
    if ((buf == NULL) || (out == NULL)) {
        return false;
    }
    if (len < 5u) {
        return false;
    }

    out->msg_id = buf[0];
    out->sensor_raw = ((uint16_t)buf[1] << 8) | (uint16_t)buf[2];
    out->battery_mv = ((uint16_t)buf[3] << 8) | (uint16_t)buf[4];
    return true;
}

Memory-mapped registers

A memory-mapped register is an address where a C load or store becomes a bus transaction with hardware meaning. volatile makes the compiler preserve accesses through that volatile-qualified lvalue. It does not make a sequence atomic, and it does not mean the register behaves like RAM. 

#include <stdint.h>

typedef struct {
    volatile uint32_t CTRL;          /* read/write */
    volatile const uint32_t STATUS;  /* read-only through C */
    volatile uint32_t TXDATA;        /* write data */
    volatile const uint32_t RXDATA;  /* read data */
} UartRegs;

#define UART0_BASE_ADDR (UINT32_C(0x40000000))
#define UART0 ((UartRegs *)(uintptr_t)UART0_BASE_ADDR)

#define UART_CTRL_TX_EN_Msk     (UINT32_C(1) << 0)
#define UART_STATUS_TX_RDY_Msk  (UINT32_C(1) << 5)

static void uart_enable_tx(void)
{
    UART0->CTRL |= UART_CTRL_TX_EN_Msk;
}

Register access traps

Trap Why it fails Better habit
REG |= MASK on write-one-to-clear flags Read-modify-write may write back flags you did not intend to clear. Write the exact clear mask required by the reference manual.
Reading a write-only register Read may be undefined, stale, or have side effects. Use shadow variables only when the hardware manual supports that model.
Ignoring reserved bits Writing reserved bits can trigger undocumented behavior. Use reset values, masks, vendor definitions, and manual-prescribed sequences.
Assuming bit-band/set/clear exists Not all MCUs provide atomic alias registers or bit-band regions. Use target-specific atomic set/clear registers where provided.
Using home-made register maps without checking vendor headers Offset, width, and access permission mistakes are easy. Prefer CMSIS/device headers or SVD-generated headers where available.

Volatile, atomics, RTOS, DMA, and barriers

This is the section to be strict about. Many firmware bugs survive because volatile is used as a superstition. It has a legitimate job, but that job is narrower than people think.

Hard distinction

volatile controls compiler treatment of accesses to volatile-qualified objects. Synchronization controls visibility, ordering, ownership, and atomicity between execution contexts. These are related in embedded systems, but not interchangeable.

Situation Use Reason
Memory-mapped register volatile register access Each access has hardware meaning.
Task waits for ISR event RTOS task notification, semaphore, queue, or event group Scheduler-aware synchronization and wakeup.
Two tasks share mutable data Mutex, queue ownership transfer, critical section Prevents concurrent mutation and documents ownership.
Bare-metal ISR flag Platform-critical section or atomic where supported Prevents lost updates and inconsistent multi-field state.
DMA reads/writes memory Cache maintenance + memory barriers + aligned buffers as required DMA is outside the C abstract machine and may not see CPU cache state.
Compiler reordering around inline asm Compiler barrier such as a memory clobber Conservatively tells compiler memory may have changed.
CPU/device ordering Architecture-specific barriers such as DMB/DSB/ISB on Arm Compiler barriers do not stop CPU-level speculation or hardware reordering.

Bad and better ISR-to-main patterns

/* Bad if more than one context writes or reads related state. */
volatile bool rx_ready;
uint8_t rx_buf[64];
size_t rx_len;

/* Better shape: protect the state transfer. */
bool app_take_rx(uint8_t *dst, size_t cap, size_t *used)
{
    if ((dst == NULL) || (used == NULL)) {
        return false;
    }

    ENTER_CRITICAL();
    bool ready = rx_ready;
    size_t len = rx_len;

    if (ready && (len <= cap)) {
        for (size_t i = 0; i < len; ++i) {
            dst[i] = rx_buf[i];
        }
        rx_ready = false;
    }
    EXIT_CRITICAL();

    if (!ready || (len > cap)) {
        return false;
    }

    *used = len;
    return true;
}

The exact correct primitive depends on interrupt priority rules, RTOS port rules, cache configuration, memory type, and whether there is more than one core.

Lifetime, storage, stack, heap, and linker sections

Embedded C is not only about type width. Where the object lives can matter more than its type.

Storage/lifetime Use Embedded limit
Automatic local Temporary function data on the stack. Stack is finite. Large local arrays can cause rare, catastrophic overflow.
Function parameter Caller-provided values and addresses. Pointer parameters do not carry ownership or length unless documented.
File-scope static Private module state and helpers. Persists for program lifetime; must be initialized/reset consciously.
External global Rare system-level shared object. Creates coupling and synchronization hazards.
Heap allocation Dynamic objects where the system design allows it. Fragmentation, failure paths, nondeterminism, and test burden.
Linker section DMA buffers, bootloader tables, flash constants, retention RAM. Compiler attributes are non-standard; linker script and startup code must agree.
Thread-local Per-thread state in hosted or RTOS-supported environments. Not universally supported in bare-metal toolchains and can increase per-thread RAM.

Stack discipline

/* Risky on small MCU stack. */
void f(void)
{
    uint8_t huge_temp[4096];
}

/* Better: static private buffer, caller-provided buffer,
   or a designed memory pool. */
static uint8_t module_workbuf[4096];
Do not hide memory ownership. If a function keeps a pointer after returning, its API must say so. Most embedded pointer bugs are lifetime bugs wearing type syntax.

Firmware API design

A good embedded C API makes the correct use boring and the wrong use obvious.

Preferred parser shape

typedef enum {
    OBC_OK = 0,
    OBC_ERR_NULL,
    OBC_ERR_SHORT,
    OBC_ERR_BAD_SOF,
    OBC_ERR_BAD_LEN,
    OBC_ERR_BAD_CRC,
    OBC_ERR_OVERFLOW
} ObcStatus;

ObcStatus obc_parse_frame(const uint8_t *frame,
                          size_t frame_len,
                          ObcPacket *out);

API rules

  • Every buffer pointer has a length or capacity.
  • Every output pointer is checked for NULL before use.
  • Every public function says whether it can block, sleep, spin, allocate, or be called from an ISR.
  • Every module hides private helpers with file-scope static.
  • Every enum crossing a wire/storage boundary is explicitly encoded as a fixed-width integer.
  • Every unit is in the name: timeout_ms, freq_hz, voltage_mv, temp_mdeg_c.
  • Every shared object has a synchronization rule next to it.

Callback pattern

typedef void (*UartRxCallback)(void *context,
                               const uint8_t *data,
                               size_t len);

typedef struct {
    UartRxCallback on_rx;
    void *context;
} UartCallbacks;

The callback receives a context pointer so it does not need hidden globals. The driver documentation must state whether the callback runs in ISR context and whether data remains valid after the callback returns.

Worked example: parser + scaling + register boundary

This example combines the toolbox: exact-width wire fields, size_t lengths, explicit endian decode, fixed-point units, float conversion for algorithm-side readability, and register access separated from packet parsing. 

#include <stdbool.h>
#include <stddef.h>
#include <stdint.h>
#include <limits.h>

enum {
    OBC_SOF0 = 0xA5u,
    OBC_SOF1 = 0x5Au,
    OBC_MIN_FRAME_LEN = 7u,
    OBC_MAX_PAYLOAD = 32u
};

typedef enum {
    OBC_OK = 0,
    OBC_ERR_NULL,
    OBC_ERR_SHORT,
    OBC_ERR_BAD_SOF,
    OBC_ERR_BAD_LEN,
    OBC_ERR_BAD_CHECKSUM
} ObcStatus;

typedef struct {
    uint8_t msg_id;
    uint16_t adc_count;      /* raw 12-bit ADC in a 16-bit field */
    uint16_t battery_mv;     /* scaled fixed-point physical unit */
    float battery_v;         /* algorithm/logging convenience */
} ObcTelemetry;

static uint16_t read_be16(const uint8_t b[2])
{
    return ((uint16_t)b[0] << 8) | (uint16_t)b[1];
}

static uint8_t checksum8(const uint8_t *bytes, size_t len)
{
    uint8_t sum = 0u;
    for (size_t i = 0u; i < len; ++i) {
        sum = (uint8_t)(sum + bytes[i]);
    }
    return sum;
}

static uint16_t adc12_to_mv(uint16_t adc_count)
{
    if (adc_count > 4095u) {
        adc_count = 4095u;
    }

    uint32_t mv = ((uint32_t)adc_count * 3300u + 2047u) / 4095u;
    return (uint16_t)mv;
}

ObcStatus obc_decode_telemetry(const uint8_t *frame,
                               size_t frame_len,
                               ObcTelemetry *out)
{
    if ((frame == NULL) || (out == NULL)) {
        return OBC_ERR_NULL;
    }

    if (frame_len < OBC_MIN_FRAME_LEN) {
        return OBC_ERR_SHORT;
    }

    if ((frame[0] != OBC_SOF0) || (frame[1] != OBC_SOF1)) {
        return OBC_ERR_BAD_SOF;
    }

    const uint8_t msg_id = frame[2];
    const uint8_t payload_len = frame[3];

    if (payload_len != 2u) {
        return OBC_ERR_BAD_LEN;
    }

    const size_t expected_len = 4u + (size_t)payload_len + 1u;
    if (frame_len != expected_len) {
        return OBC_ERR_BAD_LEN;
    }

    const uint8_t expected = checksum8(&frame[2], 2u + (size_t)payload_len);
    const uint8_t actual = frame[expected_len - 1u];

    if (actual != expected) {
        return OBC_ERR_BAD_CHECKSUM;
    }

    const uint16_t adc = read_be16(&frame[4]);
    const uint16_t mv = adc12_to_mv(adc);

    out->msg_id = msg_id;
    out->adc_count = adc;
    out->battery_mv = mv;
    out->battery_v = (float)mv / 1000.0f;

    return OBC_OK;
}

ObcTelemetry *outCaller-owned output storageParser writes after validation.

Summary and checklist

The embedded C declaration checklist

  1. Role: data, size, address, output storage, state, physical value, hardware access, or synchronization boundary?
  2. Range: what are the valid min/max values, and what happens at the boundary?
  3. Representation: is the bit width externally visible in a packet, register, flash image, or ABI?
  4. Signedness: are negative values meaningful? Are you mixing signed and unsigned values?
  5. Units: does the name say ms, hz, mv, mdeg_c, ticks, or bytes?
  6. Lifetime: stack, static, heap, linker section, caller-owned, or persistent borrowed pointer?
  7. Layout: can padding, alignment, endian, or bit-field order leak across a boundary?
  8. Concurrency: who else can touch it: ISR, task, DMA, other core, hardware?
  9. Toolchain: does this depend on compiler flags, ABI, FPU, library configuration, or vendor extensions?
  10. Failure: if this goes wrong, can the caller detect and recover?

Final rule

Do not ask “what C type do I normally use?” Ask: what contract am I writing between the code, the compiler, the hardware, and the caller?
