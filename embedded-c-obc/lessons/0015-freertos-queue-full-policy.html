Lesson 0015: FreeRTOS Queue Full Policy

A full queue is not just an API inconvenience. In flight code it is a declared failure mode: choose the timeout, count drops, and latch a fault when overload becomes unsafe.

The Embedded Problem

The IMU queue is intentionally small. If the flight task falls behind, the sensor task must not pretend every sample was delivered.

OBC habit: queue overflow is telemetry and health information. Count it, expose it, and decide when it becomes a fault.

Policy Boundary

Separate the plain-C policy from the FreeRTOS call. The policy can be tested without a scheduler; the adapter still uses the real queue API.

typedef enum
{
    IMU_DROP_POLICY_OK,
    IMU_DROP_POLICY_DROPPED,
    IMU_DROP_POLICY_FAULT
} ImuDropPolicyResult;

typedef struct
{
    uint32_t sent_sample_count;
    uint32_t dropped_sample_count;
    uint32_t drop_fault_threshold;
    bool fault_latched;
} ImuDropPolicy;

Adapter Shape

ImuDropPolicyResult imu_queue_send_with_policy(
    ImuQueueOverflow *runtime,
    const ImuSample *sample,
    TickType_t wait_ticks)
{
    BaseType_t result =
        xQueueSend(runtime->sample_queue, sample, wait_ticks);

    if (result == pdPASS)
    {
        return imu_drop_policy_record_send_ok(&runtime->drop_policy);
    }

    return imu_drop_policy_record_send_failed(&runtime->drop_policy);
}

The queue API tells you whether the send worked. Your module decides what that means for health.

Practice

Open exercises/0015-freertos-queue-full-policy/. First make the plain-C drop policy pass, then complete the real FreeRTOS adapter.

  1. Initialize counters, threshold, and fault latch.
  2. Record successful sends without clearing a latched fault.
  3. Record failed sends by incrementing dropped_sample_count.
  4. Latch a fault when the drop count reaches the configured threshold.
  5. Call xQueueSend in the adapter and feed its result into the policy.
  6. Run rtk make test, rtk make freertos-syntax, then rtk make freertos-test.
