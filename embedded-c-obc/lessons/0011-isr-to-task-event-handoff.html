Lesson 0011: ISR To Task Event Handoff

Before using FreeRTOS APIs, get the ownership rule right: interrupt code should report a small event, and one task should own the flight state machine.

The One Thing

An ISR-to-task handoff separates fast interrupt capture from normal task-level decision logic. The ISR posts a named event. The task receives it and dispatches the FSM.

OBC habit: keep ISRs short, non-blocking, and boring. Do not let an ISR mutate the flight FSM just because it noticed something important.

Shape Before FreeRTOS

This lesson uses a tiny host-testable bridge instead of real FreeRTOS. The shape is the point: later, the bridge can become a task notification or queue, but the ownership rule stays the same.

void sensor_isr(void)
{
    (void)event_bridge_post_from_isr(&bridge,
                                     FLIGHT_EVENT_APOGEE_DETECTED);
}

void flight_task_step(void)
{
    FlightEvent event;
    if (event_bridge_take_for_task(&bridge, &event))
    {
        (void)flight_fsm_dispatch(&fsm, event);
    }
}

Who May Do What?

CodeAllowedNot allowed in this pattern
ISR sideCapture the event and post it to the bridge.Block, allocate, run long policy code, or mutate the FSM.
Task sideTake pending events and dispatch the FSM one at a time.Assume every event is present forever; consume means consume.
FSM coreApply transition policy as ordinary C.Know whether the event came from an ISR, queue, test, or replay.
Review smell: if both an ISR and a task write fsm.phase, ownership is already unclear.

Overflow Is A Policy

A one-slot bridge can already contain a pending event. If a new event arrives before the task consumes the old one, the code must choose a policy deliberately.

PolicyWhen it can make senseRisk
Reject newestThe first event must be handled exactly once.Later urgent events may be dropped.
Overwrite oldestOnly the latest sample/event matters.A real transition event can disappear.
Use a queueMultiple events must be preserved in order.More memory and a full-queue policy still exists.

The exercise uses reject newest and count drops. That makes lost events visible in tests.

FreeRTOS Preview

In real FreeRTOS, the bridge would usually be a task notification for a compact signal or a queue when event values must be buffered. ISR code uses the FromISR family of APIs where the port requires it.

/* Later shape, not used in this exercise. */
xQueueSendFromISR(flight_event_queue, &event, &higher_priority_task_woken);

/* Task side owns the FSM. */
if (xQueueReceive(flight_event_queue, &event, timeout_ticks) == pdPASS)
{
    (void)flight_fsm_dispatch(&fsm, event);
}
