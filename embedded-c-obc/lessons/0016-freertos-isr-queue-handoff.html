Lesson 0016: FreeRTOS ISR Queue Handoff

When an IMU interrupt says "a sample is ready", the ISR should hand off the sample and leave. The flight task owns the real decision logic.

The Embedded Problem

An IMU DMA-complete interrupt fires while the scheduler is running. The ISR can copy a compact sample into a queue, but it should not run the flight FSM, block on a queue, or do long health-policy work.

OBC habit: interrupt context detects and hands off. Task context decides.

The Real API Shape

void IMU_DMA_IRQHandler(void)
{
    ImuSample sample = read_dma_sample_registers();
    BaseType_t higher_priority_task_woken = pdFALSE;

    if (xQueueSendFromISR(sample_queue,
                          &sample,
                          &higher_priority_task_woken) != pdPASS)
    {
        isr_drop_count++;
    }

    portYIELD_FROM_ISR(higher_priority_task_woken);
}

xQueueSendFromISR is the ISR-safe queue send. The third argument lets FreeRTOS tell you whether the send woke a task that should run as soon as the ISR exits.

Ownership Split

PlaceOwnsDoes not own
ISRShort hardware acknowledgment, compact sample handoff, simple drop counter.Flight phase transitions, blocking waits, queue draining, printf-style logging.
QueueOrdered sample storage by copy.Health meaning, state-machine transitions, dynamic routing decisions.
Flight taskBlocking receive, sample validation, policy modules, FSM dispatch.Hardware interrupt timing.

Why The Yield Argument Matters

If the flight task was blocked on the queue and has higher priority than the interrupted task, the send can make it ready. The ISR records that in xHigherPriorityTaskWoken, then portYIELD_FROM_ISR requests a context switch at the proper interrupt-exit point.

Review smell: calling normal task APIs from an ISR, or ignoring the higher-priority wake value, usually means the interrupt boundary was not designed deliberately.

Quick Check
