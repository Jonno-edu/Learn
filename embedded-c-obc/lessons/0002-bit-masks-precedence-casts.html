Lesson 0002: Masks Before Meaning

Lesson 1 exposed the real gap: not firmware intent, but C expression mechanics. This lesson teaches the expression pattern behind status bits, packet fields, and register-width-to-byte conversions.

The One Thing

When a wider integer contains several independent fields, never compare the whole value unless the whole value is the contract. First isolate the field with a mask. Then compare or cast the isolated result.

Driver habit: (value & MASK) == MASK checks whether the named bits are set while ignoring unrelated bits.

The Precedence Trap

This expression looks plausible: 

status & RX_READY_MASK == RX_READY_MASK

But C groups equality before bitwise AND here, so the compiler reads it as: 

status & (RX_READY_MASK == RX_READY_MASK)

The fix is not memorizing the whole precedence table. The fix is to parenthesize the bit operation every time you compare it: 

(status & RX_READY_MASK) == RX_READY_MASK

Whole Register Equality Fails

Suppose bit 0 means RX ready and bit 2 means overrun. If the status is 0b00000101, RX is ready and overrun is also set. 

status == RX_READY_MASK        /* false: status is 5, mask is 1 */
(status & RX_READY_MASK) != 0 /* true: bit 0 is set */

Explicit Narrowing

A register or packet word may be uint32_t, while the field you return is only a byte. Mask first, then cast: 

return (uint8_t)(word & LOW_BYTE_MASK);

The cast is not magic. It is documentation for reviewers: after the mask, losing upper bits is intentional.

Quick Check

1. Which expression checks bit 3 while ignoring every other bit?

2. Why write (uint8_t)(word & 0xffu) instead of just returning word?
