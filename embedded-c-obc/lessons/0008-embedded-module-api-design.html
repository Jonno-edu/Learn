Lesson 0008: Embedded Module API Design

A module API is the boundary between code that uses a capability and code that owns its details. This lesson prepares you for a state machine by designing a small recovery decision module first.

The One Thing

In embedded C, a good module API makes the safe path obvious: callers see the types, constants, and functions they are allowed to use, while private state and helper logic stay inside the .c file.

OBC habit: the header is a contract. Put only the caller's promises and the module's promises there. Hide everything else.

Public Versus Private

Belongs in .h Belongs in .c
Public types callers must name, such as RecoveryGateDecision. Private helper functions, lookup tables, thresholds, and implementation details.
Function prototypes callers need, such as recovery_gate_update(...). static functions and file-scope objects used only by this module.
Small config or input structs when they make call sites clearer. Validation details, transition logic, debounce counters, and derived calculations.

This separation is why static at file scope matters: it gives a function or object internal linkage, so it is private to that translation unit.

A Small Recovery Gate API

Before writing a full state machine, design a smaller decision module. It receives one snapshot of flight data and returns one decision. That gives you API practice without event-history complexity. 

typedef enum {
    RECOVERY_GATE_HOLD = 0,
    RECOVERY_GATE_DEPLOY_DROGUE,
    RECOVERY_GATE_DEPLOY_MAIN,
    RECOVERY_GATE_FAULT
} RecoveryGateDecision;

typedef struct {
    int32_t altitude_m;
    int32_t vertical_velocity_mps;
    FlightState flight_state;
    bool pyro_armed;
} RecoveryGateInput;

RecoveryGateDecision recovery_gate_evaluate(const RecoveryGateInput *input);

This API says: caller owns the input object, the module reads it, and the module returns a decision. No hidden buffer, no allocation, no protocol bytes, and no output parameter unless the API needs more than one result.

Design Questions

Question Default embedded answer
Does the module need memory between calls? If yes, use an explicit context struct such as RecoveryGate. If no, a pure evaluate function is simpler.
Can invalid input happen? Represent impossible states with types when you can. Validate pointer arguments and out-of-range values when you cannot.
Should the function return a status or the result? Return the main result when failure can be represented clearly. Use a status plus output parameter when failure and result are separate.
How do tests reach private logic? Prefer testing through the public API. If private logic needs direct tests, the private function may be the wrong boundary.

State Machine Preparation

A state machine module uses the same boundary rules, but it usually owns state: 

typedef struct {
    FlightPhase phase;
    uint32_t samples_in_phase;
    bool fault_latched;
} FlightStateMachine;

void fsm_init(FlightStateMachine *fsm);
FsmResult fsm_update(FlightStateMachine *fsm, const FsmInput *input);

The caller provides storage for FlightStateMachine. The module initializes and mutates it. This avoids dynamic allocation and makes tests deterministic.

Review smell: if every other file can mutate your module's internal fields directly, the module does not really own its state.
