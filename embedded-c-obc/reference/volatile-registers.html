C volatile And Registers

Use volatile when the value can change for reasons the compiler cannot see, such as a memory-mapped peripheral register. It controls compiler optimization of accesses. It is not a lock, not an atomic operation, and not a FreeRTOS synchronization primitive.

Use It For

  • Memory-mapped hardware registers.
  • Register structs whose members represent hardware locations.
  • Objects touched by C signal handlers in hosted C, with extra rules.

Do Not Use It As

  • A mutex.
  • A replacement for atomics.
  • A FreeRTOS ISR-to-task signaling mechanism.
  • A fix for every optimization-related bug.

Register Pattern

#include <stdint.h>

#define UART0_BASE_ADDR 0x40001000u

typedef struct {
    volatile uint32_t STATUS;
    volatile uint32_t DATA;
} UartRegs;

#define UART0 ((UartRegs *)UART0_BASE_ADDR)

enum {
    UART_STATUS_RX_READY = 1u << 0
};

uint8_t uart0_read_byte(void)
{
    while ((UART0->STATUS & UART_STATUS_RX_READY) == 0u) {
        /* bounded timeout should be added in real flight code */
    }

    return (uint8_t)(UART0->DATA & 0xffu);
}

Pointer Qualifier Reading

Declaration Meaning
volatile uint32_t *p The object pointed to is volatile. The pointer variable can change.
uint32_t * volatile p The pointer variable is volatile. The object pointed to is not volatile.
volatile uint32_t * const p The object pointed to is volatile. The pointer variable cannot be reassigned.

Flight-Firmware Heuristic

If the variable represents hardware, ask "should every load/store reach the bus?" If yes, it likely belongs behind a volatile register abstraction. If it represents communication between tasks or ISRs, ask "what synchronizes ownership and ordering?" The answer should usually be a FreeRTOS primitive, critical section, queue, semaphore, notification, or carefully reviewed atomic pattern.
