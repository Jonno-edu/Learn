Embedded Module API Design

A quick reference for designing small C module boundaries before building larger state machines.

Header Rule

The .h file is the public contract. Put caller-visible types and function prototypes there. Keep private helpers, thresholds, and file-local state in the .c file.

Common API Shapes

Shape Use when
Decision evaluate(const Input *in) The module does not need memory between calls and failure can be represented as a decision.
bool build(const Input *in, Output *out) The function can fail and must produce a separate output object.
void init(Module *m)
Result update(Module *m, const Input *in)		 The module owns state across calls, such as a state machine, filter, or debouncer.

State Ownership

typedef struct {
    FlightPhase phase;
    uint32_t samples_in_phase;
    bool fault_latched;
} FlightStateMachine;

void fsm_init(FlightStateMachine *fsm);
FsmResult fsm_update(FlightStateMachine *fsm, const FsmInput *input);

Avoid hidden global state by default. Caller-provided context structs make ownership explicit, keep storage deterministic, and make host-side tests easier.

Review Checklist

  • Can a caller understand the module by reading only the header?
  • Are implementation-only helpers marked static?
  • Does the API say who owns each object?
  • Are invalid inputs handled consistently?
  • Can behavior be tested through the public API?
