FreeRTOS OBC Toolbox Decision Table

Choose the primitive that matches the boundary. Avoid using a more powerful primitive to compensate for unclear ownership.

ProblemLikely toolWatch out for
Wake one known task with compact flags.Task notification.Repeated identical flags coalesce.
Preserve every sample or command in order.Queue.Queue depth, item size, overflow policy.
Send from interrupt context.FromISR API.Blocking, long work, missing yield request.
Run at a fixed period.vTaskDelayUntil.Overrun and freshness visibility.
Protect shared mutable state.Single owner first; mutex only if sharing remains.Priority inversion, hidden coupling, lock ordering.
Represent multiple independent condition bits.Event group, later.Global condition soup and unclear ownership.
Move byte streams or variable-length messages.Stream/message buffer, later.Framing, bounds, and backpressure.
Short uninterruptible critical update.Critical section, later.Blocking interrupts too long.

Review Sequence

  1. Name the owner of the state.
  2. Name the execution context.
  3. Name what crosses the boundary.
  4. Choose the smallest primitive that preserves what matters.
  5. Define timeout, overflow, or stale-data behavior.

If two tasks both mutate the same struct, pause before adding a mutex. The better design may be to make one task own the struct and send it messages.
