FreeRTOS Queue Full Policy

A reference for turning queue send failures into visible health behavior.

Basic Pattern

BaseType_t result =
    xQueueSend(queue, &sample, wait_ticks);

if (result == pdPASS)
{
    sent_sample_count++;
}
else
{
    dropped_sample_count++;
    if (dropped_sample_count >= drop_fault_threshold)
    {
        fault_latched = true;
    }
}

Timeout Choices

ValueMeaningCommon policy
(TickType_t)0Do not wait for space.Best for explicit drop counting in a fixed-rate producer.
Finite ticksWait briefly.Useful if a small delay is acceptable.
portMAX_DELAYWait indefinitely when the configuration supports it.Usually avoid for periodic sensor sends unless that stall is intentional.

A full queue can mean the consumer task is late, stuck, lower priority than expected, or processing too much work per item. Do not hide that with an unbounded wait unless the system design says the producer may stop there.

Review Checklist

  • Is the queue depth intentionally chosen?
  • Is the send block time explicit?
  • Is every non-pdPASS send counted?
  • Does the health interface expose dropped_sample_count?
  • Does repeated overflow latch a fault until a deliberate reset path clears it?
  • Are pointer and queue-handle checks done before dereference or API use?
